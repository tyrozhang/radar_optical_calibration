解决方案：创建基于真实速度的航迹预测器 `TrackPredictor.java`

文件位置：`src/main/java/gds/cloud/module/antidrone/utils/radar/calibration/TrackPredictor.java`

- 雷达约2秒更新一次航迹（位置+速度）
- 光电50ms查询一次预测角度，实现平滑跟踪
- 支持直接传入速度数据（velEast/North/Up）
- 也支持用位置差分估算速度

## 重构：抽取 TrackingCore 共享基础类

问题：RadarOpticTrackerV2 和 SmoothTracker 存在代码重复（角度计算、雷达补偿查找、归一化），且 SmoothTracker 反向依赖 V2 的内部类型

方案A实施：保留两个类，抽取共享逻辑到 TrackingCore

### 新增文件
- `TrackingCore.java` — 共享基础类（纯静态方法）
  - `RadarCompensation` 接口（非 sealed，允许外部实现）
  - `FixedCompensation` / `SegmentedCompensation` 接口
  - `normalizeAz()` / `normalizeEl()` — 角度归一化
  - `getRadarCompensation()` — 雷达补偿查找
  - `computeAngles()` — 基础角度计算（几何角+偏差+补偿+归一化）

### 改动文件
- `RadarOpticTrackerV2.java`
  - `compute()` 委托 TrackingCore.computeAngles()，保留调试打印/radarArea/丰富字段
  - `RadarCompensation` sealed interface extends `TrackingCore.RadarCompensation`
  - `FixedCompensation` / `SegmentedCompensation` record 同时实现 V2 和 TrackingCore 接口
  - **对外 API 100% 不变**：类型名、TrackingCommand、compute() 签名
- `SmoothTracker.java`
  - radarComp 类型从 `RadarOpticTrackerV2.RadarCompensation` 改为 `TrackingCore.RadarCompensation`
  - `computeAngles()` / `getRadarCompensation()` / `normalizeAz()` / `normalizeEl()` 委托 TrackingCore
  - 消除了对 RadarOpticTrackerV2 的反向依赖（测试中仍可用 V2.FixedCompensation）

### 验证
- 全部 Java 文件编译通过（-encoding UTF-8）
- SmoothTracker 4个测试通过
- CalibrationValidator 结果不变（Az RMS 2.5572°, El RMS 0.2279°）
- TrackerInvokerTest 通过

## SmoothTrackingController 实现（新增）

封装"新点立即执行+后续预测"的控制逻辑，解决"每个雷达点先执行原始位置再做预测"的需求。

### 核心逻辑
- **新点窗口**（100ms）：用雷达原始位置转动光电
- **预测阶段**：基于最新两点速度外推预测（仅当 `tracker.isReady()` 为 true）
- **缓冲期**（下一雷达点前50ms）：等待新数据

### 文件位置
`src/main/java/gds/cloud/module/antidrone/utils/radar/calibration/SmoothTrackingController.java`

### 关键设计
- `tick()` 现为私有方法，仅由 `start()` 启动的内部循环调用，统一使用 `System.currentTimeMillis()`
- 预测阶段严格检查 `tracker.isReady()`，确保只有有完整轨迹时才预测

### 测试验证
通过5个逻辑判定：
1. 每个雷达点都有立即执行（新点窗口内）
2. P0周期只有立即执行无预测（单点无法预测，isReady()=false）
3. P1/P2周期有立即执行+多次预测（约93次/周期）
4. P2预测Az < P1预测Az（目标远离，角度减小）
5. 滑动窗口后 t1=P1, t2=P2
