## 二、原始报告未覆盖的新发现

### 🔴 新发现 — 阻断级（3项）

#### N1 — 标定分段距离基准与在线查找距离基准不一致

**位置**：
- 标定时：`CalibrateV2.java:74` — 使用 **雷达站→目标** 的距离
- 在线时：`TrackingCore.java:118` — 使用 **光电站→目标** 的距离

```java
// 标定侧 (CalibrateV2.java:74)
double distKm = CoordinateUtils.distance3D(radarStation, p.targetBlh()) / 1000.0;

// 在线侧 (TrackingCore.java:118)
double dist = CoordinateUtils.haversineDistance(
    targetB, targetL, opticalBlh.B(), opticalBlh.L());
```

**影响**：雷达站与光电站通常不共址（间距数十至数百米）。同一目标可能在标定时落入段A、在线时落入段B，导致查到的补偿值与标定值不匹配。**这是标定-补偿闭环的逻辑缺陷**。

#### N2 — 控制线程非 daemon 导致 JVM 无法正常退出

**位置**：`SmoothTrackingController.java:185-190`

```java
new Thread(() -> {
    while (running) {
        tick();
        sleep(predictionInterval);
    }
}, "TrackingController").start();  // 非 daemon 线程
```

`stop()` 仅设置 `running = false`，但非 daemon 线程会阻止 JVM 退出，直到线程自然结束（最多 `predictionInterval` 毫秒延迟）。在异常关闭场景下可能导致进程残留。

#### N3 — 空雷达数据静默产出无效标定结果

**位置**：`CalibrateV2.java:71-88`

当所有数据点的 `radarBlh` 均为 null 时，`errorSamples` 为空列表，`RadarErrorAnalyzer.analyze()` 返回全零统计。标定程序正常完成，输出补偿值为 0 的标定文件，**无任何警告**。

### 🟡 新发现 — 重要级（7项）

#### N4 — SmoothTracker 与 TrackPredictor 双轨并行

两个类都做"基于速度外推预测目标位置"，但互不调用，功能重叠：

| 维度 | TrackPredictor | SmoothTracker |
|------|---------------|---------------|
| 速度表示 | m/s (East/North/Up) | °/s (dLat/dLon/dH) |
| 速度来源 | 雷达直接提供 或 位置差分 | 仅位置差分 |
| 机动检测 | ✅ 有 | ❌ 无 |
| 外推时长保护 | ✅ maxPredictionSeconds=2.0 | ❌ 无限制 |
| 速度滤波 | ❌ 无 | ❌ 无 |
| 时延补偿 | ❌ 不支持 | ✅ totalDelayMs |

#### N5 — CalibrateV2 和 TrackingCore 角度归一化范围不统一

```java
// CalibrateV2.java:188-193 — 归一化到 [-180, 180)
// TrackingCore.java:77-80  — 归一化到 [0, 360)
```

标定输出的 `dAz0` 在 [-180, 180)，在线跟踪将其与 [0, 360) 的几何角相加。数学上正确但调试困难。

#### N6 — CoordinateUtils.ecefToBlh() Bowring 公式分母幂次错误

`Simulator.java:234-266` 重复了 Bowring 公式和 ENU→ECEF 旋转矩阵（约30行），但更严重的是**两处 Bowring 公式存在数学错误**：

```java
// Simulator.blhFromEnu() (line 253) — 正确 ✅
double denom = Math.pow(1.0 - CoordinateUtils.E2 * sinTheta2, 1.5);  // 幂次 3/2

// CoordinateUtils.ecefToBlh() (line 55) — 错误 ❌
// 分母项 (1 - E2 * sinTheta * sinTheta)  ← 幂次为 1，应为 3/2
double B_rad = Math.atan2(
    Z + (E2 * (1 - E2) * A / (1 - E2 * sinTheta * sinTheta)) * sinTheta * sinTheta * sinTheta,
    p - E2 * A / (1 - E2 * sinTheta * sinTheta) * cosTheta * cosTheta * cosTheta);
```

标准 Bowring (1976) 迭代公式的分母项为 `(1 - e²·sin²θ)^{3/2}`，CoordinateUtils 用了 **1 次方**而非 **3/2 次方**。这并非"等效公式的不同精度路径"，而是**数学公式错误**。由于 `e² ≈ 0.0067`，实际偏差极小（近地目标约毫米级），但将公式错误定性为"精度差异"会误导修复方向——修复者可能以为只需统一实现风格，而实际需要改正公式。

#### N7 — 默认目标高度 0.01m 物理上不合理

```java
// TrackingCore.java:29
public static final double DEFAULT_TARGET_HEIGHT = 0.01;  // 1cm
```

对无人机目标，1cm 高度不现实。可能影响光电设备的目标框选和视场计算。

#### N8 — valid_until 字段写入但从未检查

`CalibrateV2.saveCalibrationV2()` 写入 `valid_until`（当前日期+365天），但 `TrackerConfigLoader.load()` 不解析也不检查。过期标定参数可被无警告使用。

#### N9 — SmoothTracker 不处理时间戳倒退

如果雷达因时钟跳变发送早于 `t2Time` 的时间戳，`computeVelocity()` 有零速保护（`dt <= 0 → [0,0,0]`），但数据已污染，后续外推基于错乱的 t1/t2 顺序。

#### N10 — SmoothTracker 内插可能在查询时间 < t1Time 时变为无保护外推

```java
// SmoothTracker.java:195
double ratio = 1.0 + dt / totalDt;  // 当 queryTime < t1Time 时 ratio < 0
```

当 `queryTime < t1Time` 时，ratio 变为负值，内插实际变为向更早时间的外推，无保护。

| **N11** | `Simulator.printSegmentStats` 硬编码分段边界，与类顶部 `DISTANCE_SEGMENTS` 常量重复定义 |

## 四、架构层面问题

1. **线程安全缺陷深及多层**（P9 扩展）：不仅 Controller 层 `onRadarData()/tick()/reset()` 存在竞态，`SmoothTracker` 内部 `update()/getCommand()` 亦跨线程无同步，修复需下沉至 Tracker 内部
2. **SmoothTracker 与 TrackPredictor 的双轨并行**（N4）：两个模块功能高度重叠但不共享代码，存在维护负担和功能不对等
3. **全项目无自动化测试**（N14）：所有测试依赖人工查看控制台输出，无法集成 CI
4. **角度归一化范围不统一**（N5）：标定用 [-180,180)，在线用 [0,360)
5. **坐标转换公式错误**（N6）：`CoordinateUtils.ecefToBlh()` Bowring 公式分母幂次为 1 而非 3/2，虽实际偏差极小但属于数学公式错误
6. **JSON 手工拼接分散在多个类中**（N12）：缺乏统一的序列化层，维护困难
7. **三个 SimulatedPoint 定义并存**（N13）：数据流转中存在语义歧义风险
